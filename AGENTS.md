# AGENTS.md

## 项目概述

本项目是一个基于 Python 的 AI 旅游助手，目标是构建一个具备真实工程结构的 AI 应用，而不是简单的问答 demo。当前项目重点是 FastAPI 接口、本地旅游知识库、RAG 检索增强生成，以及后续逐步加入 Agent Workflow 能力。

项目目前处于学习和 MVP 阶段，应优先保证代码结构清晰、RAG 行为稳定、回答可解释、资料不足时能够拒答。不要为了追求技术栈完整而过早引入 Milvus、Kubernetes、Neo4j、Elasticsearch、微服务等重型基础设施。

## 当前技术方向

项目后续应按照以下顺序演进：

1. FastAPI 接口层。
2. 本地旅游知识库。
3. 基于 FAISS 的向量检索。
4. BM25 + FAISS 混合检索。
5. Reranker 模型重排。
6. 检索置信度判断。
7. 回答中的来源引用。
8. 常见旅游问题的 Eval 测试。
9. 在 RAG 流程稳定后，再引入 Agent Workflow 编排。

当前优先级不是 Multi-Agent，而是先把单 Agent 的 RAG 流程做可靠。

## RAG 行为规则

修改 RAG 逻辑时，需要遵守以下规则：

* 必须区分本地知识库问题和实时信息问题。
* 如果知识库中没有足够依据，助手必须说明“根据当前资料，我无法确认”或表达同等含义。
* 当检索置信度较低时，不允许让 LLM 编造答案。
* 只要答案基于检索资料生成，就应尽量提供来源信息。
* 检索结果进入 Prompt 之前，应经过过滤、重排和置信度判断。
* 不要把噪声内容、重复 chunk、无关资料直接塞进最终 Prompt。

对于“今天几点开门”“当前票价”“实时天气”“交通状态”等实时问题，系统不能仅根据静态本地知识库硬答。除非已经接入实时工具，否则应明确说明当前资料无法确认。

## 工程原则

修改代码时，应优先做小而确定的改动。除非用户明确要求重构，否则不要大规模重写已有实现。

代码应优先保证可读性，而不是追求复杂技巧。这个项目同时用于学习 AI 应用工程，所以代码应容易理解、容易解释、容易调试。

新增功能时，应保持模块化。例如 Query Analysis、检索、混合检索、重排、置信度判断、Prompt 构建、LLM 调用、答案生成等逻辑，应该尽量拆分到不同模块中，不要全部混在一个大函数里。

避免过度抽象。除非有明确收益，否则不要创建复杂的类继承结构。

不要随意引入新的第三方依赖。如果确实需要新增依赖，应说明为什么需要它，以及不用它会有什么问题。

## 代码规范

使用兼容 Python 3.11+ 的写法。

新增函数时，尽量添加合理的类型注解。

函数命名要清晰，优先使用类似 `analyze_query`、`retrieve_candidates`、`rerank_chunks`、`check_retrieval_confidence`、`build_rag_prompt` 这样的名称。

函数应尽量只负责一件主要事情。不要把检索、Prompt 构建、LLM 调用和结果处理全部写在一个函数里。

错误处理要明确，不要静默吞掉异常。

不要把 API Key、密码、私密配置等硬编码到代码仓库中。

如需本地环境变量，应使用 `.env` 或类似方式管理。

## FastAPI 接口规则

FastAPI 的路由函数应保持轻量。

路由层只负责接收请求、调用服务、返回响应。复杂业务逻辑应放到 service、rag 或 core 模块中。

请求和响应结构应优先使用 Pydantic 模型。

接口返回的错误信息应清晰，便于调试。

当逻辑变复杂后，不要把检索、Prompt 拼接和 LLM 调用全部直接写在路由函数中。

## 测试与 Eval 要求

如果修改了检索、重排、置信度判断、Prompt 构建或答案生成逻辑，应同步新增或更新 Eval 测试用例。

Eval 测试集至少应覆盖以下类型：

* 正常的旅游推荐问题。
* 知识库中明确存在答案的事实问题。
* 知识库中没有答案的问题。
* 不应由静态知识库回答的实时问题。
* 用户意图模糊的问题。
* 同时涉及城市、类别、偏好的问题。

系统应满足以下预期：

* 当检索证据充分时，可以正常回答。
* 当检索证据不足时，应拒答或说明资料不足。
* 不应编造知识库中没有的事实。
* 使用本地资料回答时，应尽量引用来源。
* 回答应和用户问题保持相关，不要泛泛而谈。

## 常用命令

修改命令之前，应先检查项目中的实际文件结构。

常见命令可能包括：

```bash
python -m venv .venv
pip install -r requirements.txt
uvicorn app.main:app --reload

```

如果用户在 Windows PowerShell 中开发，应尽量使用符合 Windows 环境的命令风格，不要默认使用 Linux 或 macOS 命令。

## Git 与变更管理

修改代码前，应先查看相关文件，不要凭空猜测项目结构。

不要修改和当前任务无关的文件。

不要在没有要求的情况下格式化整个项目。

完成修改后，应说明以下内容：

* 改了什么。
* 为什么这么改。
* 如何测试。
* 是否存在风险或后续待办。

如果改动影响 RAG 流程，需要说明它影响的是检索、重排、置信度判断、Prompt 构建，还是答案生成。

## AI 应用设计偏好

本项目应体现真实 AI 应用工程能力，而不是只展示框架调用。

做设计决策时，应考虑：

* 为什么需要这个设计。
* 它是如何工作的。
* 它带来了什么取舍。
* 为什么不用更简单的方案。
* 当前项目规模是否真的需要这个工具。

对于几千条本地知识数据，优先使用 FAISS 或 Qdrant，不要直接上 Milvus。只有当数据规模、并发、分布式检索和运维需求明显上升时，才考虑 Milvus。

当前阶段应优先构建清晰可靠的 RAG Workflow，不要过早引入复杂 Multi-Agent。

固定流程适合使用 Workflow。只有当多个专业 Agent 需要独立推理、互相通信、任务分工或协作决策时，才考虑 Multi-Agent。

## Memory 与上下文规则

不要把所有 Memory 都直接塞进 system prompt。

上下文应分层管理：

1. 系统规则。
2. 项目规则。
3. 用户偏好。
4. 会话摘要。
5. 最近对话。
6. 检索资料。
7. 工具结果。

稳定且高优先级的规则可以放在 Prompt 靠前位置。检索资料必须和指令清晰分隔，不能混在一起。

每次发送给 LLM 前，都应减少上下文噪声，避免重复、过期、无关信息影响回答质量。

## 文档规则

当项目行为发生明显变化时，应同步更新 README.md。

README 中应记录重要的 RAG 设计决策，尤其是：

* 为什么使用混合检索。
* 如何判断检索置信度。
* 什么情况下系统会拒答。
* 如何展示来源引用。
* 如何运行 Eval 测试。

不要在文档中写虚假的生产化能力。如果某个功能只是实验性或尚未完成，应明确说明。

## 对用户的沟通风格

默认使用中文交流。

当前角色是编程助手，而不是单纯的知识讲解助手。回答时应优先围绕代码修改、项目推进、问题定位、工程优化和下一步行动展开。

每次完成一个任务后，必须说明本次改动带来的改进点。改进点应尽量具体，例如代码结构更清晰、检索逻辑更稳定、Prompt 更可控、异常处理更完整、可测试性更好、后续扩展更方便等。

回复用户时，应优先采用以下结构：

1. 本次完成了什么。
2. 改进了哪些地方。
3. 如何验证是否成功。
4. 是否还有后续建议。

不要只告诉用户“已经完成”，而是要让用户知道这次修改对项目质量有什么实际提升。

解释代码时不要过度展开基础 Python，除非用户明确要求。重点说明这段代码为什么要这样改、它解决了什么问题、对当前项目有什么影响。

当用户的理解部分正确时，应先肯定正确部分，再补充缺失的工程视角。不要简单地回答“对”或“不对”。

如果发现当前实现中存在潜在问题，应主动指出，并说明是否需要现在处理。不要一次性扩大修改范围，除非用户明确要求。

除代码、目录树、命令、流程图、架构图之外，正常说明应使用自然段落，不要把一句完整的话拆成很多短行。

整体沟通风格应像一名正在协助开发项目的工程师：直接、清楚、可执行，并且每次都帮助用户理解项目相比之前变好了哪里。


## 不要做的事情

不要因为某个技术热门，就随意加入 LangChain、LangGraph、Qdrant、Neo4j、Elasticsearch、Kubernetes、LangFuse、OpenTelemetry 或 DSPy。

不要在任务没有要求的情况下，把当前可运行实现替换成大型框架。

不要让 LLM 回答没有资料支撑的旅游事实。

不要忽略来源引用要求。

不要把实时信息需求和静态本地知识库混为一谈。

不要创建不必要的文件、抽象层或依赖。

## 完成标准

一个任务只有在满足以下条件时，才算完成：

* 用户要求的改动已经实现。
* 代码符合当前项目结构。
* 基本错误情况已经考虑。
* 如果影响检索或回答质量，已经更新相关测试或 Eval 用例。
* 最终回复说明了改动内容、验证方式和后续风险。
